- 仿真实验.txt

  - 实现方式：文本化整理实验叙述，作为论文段落素材。

### 2.2 `algo/mb_ppo`：MB-PPO 训练主链路

#### 2.2.1 `mb_ppo_config.py`

- 实现内容：默认配置集中管理。
- 核心实现点：
  - 训练超参：lr、gamma、gae_lambda、clip_ratio、ppo_epochs、mini_batch_size 等。
  - 结构超参：single_head、hidden_size、precoding_dim、antenna_count、csi_complex_dim。
  - 约束相关：lambda_init、lambda_lr、qos_threshold、paper_fixed_qos_threshold。
  - 场景参数：n_gts、n_ubs、n_eve、range_pos、velocity_bound、tx_power_max_dbm。

#### 2.2.2 `agents/mb_ppo_agent.py`

- 实现内容：Actor-Critic 网络（多分支/单头可切换）。
- 实现方式：
  - 输入特征拆分：GT 特征经 `gt_encoder` 提取，再与其他特征拼接进入共享 `actor_encoder`。
  - 多分支动作头：traj、precoding、power、rate 四个均值头及对应 log_std 参数。
  - 采样机制：高斯分布采样后经 tanh 压缩到 [-1, 1]，并做 tanh 修正对数概率。
  - 评估机制：`evaluate_actions` 复算 log_prob/entropy 供 PPO 更新。
  - 动作输出：`split_actions` 将拼接动作拆为环境需要的四分支字典。

#### 2.2.3 `mb_ppo_learner.py`

- 实现内容：PPO 学习器与参数更新。
- 实现方式：
  - `take_actions`：前向采样动作、保存旧策略对数概率与状态价值。
  - `update`：按 mini-batch 做 PPO clipped surrogate 更新，包含 value loss、entropy regularization、梯度裁剪。
  - `update_lambda`：对偶变量标准更新，`lambda <- max(0, lambda + lr * signal)`。
  - `save_model/load_model`：包含网络参数、优化器状态、lambda。

#### 2.2.4 `ppo_buffer.py`

- 实现内容：轨迹缓存与 GAE 计算。
- 实现方式：
  - 存储：obs_other、obs_gt、state、action、log_prob、reward、done、value、qos_signal。
  - 计算：`compute_returns_and_advantages` 反向递推 GAE。
  - 输出：`as_tensors` 转为训练张量，`iter_indices` 生成 batch 索引。
  - 约束信号：`mean_qos_signal` 作为 lambda 更新输入。

#### 2.2.5 `run_mbppo.py`

- 实现内容：训练入口与全流程 orchestration。
- 实现方式：
  - 输出目录管理：自动创建 `mb_ppo_data/exp*` 或指定 output_dir。
  - 配置装配：加载默认配置并合并 CLI 参数；支持固定 R_th 强制逻辑。
  - 环境创建：实例化训练环境与评估环境。
  - 训练循环：采样、存 buffer、PPO 更新、lambda 更新、写 TensorBoard。
  - 周期评估：`evaluate_policy` 汇总 return/QoS/EE/fairness/tx_power。
  - 数据落盘：train_returns、eval_history、episode_metrics、eval_traces。
  - baseline 开关：mbppo/ppo/circular/hover/sdma/noma。

#### 2.2.6 `utils.py`

- 实现内容：通用工具函数。
- 实现方式：参数检查、随机种子、pickle 保存读取、config JSON 序列化。

### 2.3 `algo/offpolicy_baselines`：离策略对照入口

#### `run_offpolicy.py`

- 实现内容：SAC 对照训练链路。
- 实现方式：
  - `Sb3MbPpoEnv`：把项目环境封装为 Gymnasium 接口。
  - `EpisodeMetricsCallback`：采集 episode 指标并做 lambda 对偶更新。
  - `_build_model`：构建 SAC 模型。
  - 与主链路共享配置语义和输出目录结构。

### 2.4 `mb_ppo_env`：环境与物理映射

#### 2.4.1 `mb_ppo_environment.py`

- 实现内容：环境核心实现（已集中化）。
- 实现方式：

1. 基础环境类 `UbsRsmaEvn`
   - 地图、用户、无人机初始化（`n_eve` 仅保留构造兼容，不参与保密链路计算）。
   - 距离更新、用户关联、基础观测状态构造。
   - 提供 reset/step 与基础 API，作为连续环境父类。

2. 连续动作类 `MbPpoEnv(UbsRsmaEvn)`
   - 继承基础环境，扩展为 MB-PPO 连续动作接口。
   - 动作四分支：轨迹、预编码、功率、速率。
   - `_physical_rate_mapping`：
     - 预编码：实部/虚部重构复数波束；`baseline=ppo` 时不做硬归一化约束，`baseline!=ppo` 时做 `l2` 归一化。
     - 功率：`rho` 控制总功率；硬映射模式使用 `alpha=softmax` 分配流功率；PPO 基线保留“无硬映射、靠惩罚约束”。
     - 速率：beta（softmax）分配公共速率份额。
     - 依据显式 common/private SINR 计算 common/private/effective rate（RSMA/SDMA/NOMA 可切换）。
   - 信道：保留 URA 导向矢量 + 复莱斯信道（含相位），用于 CSI 特征与速率计算。
   - baseline 行为：
     - hover：静止。
     - circular：圆轨迹。
     - ppo/mbppo/sdma/noma：按模式选择有效速率组合。
   - 奖励：
     - `reward = reward_ee_scale * EE - lambda * reward_qos_scale * qos_gap_sq_sum - reward_power_penalty_scale * power_violation`。
     - `reward_ee_scale` 在环境内部放大 1000 倍，以匹配 QoS 惩罚量级。
     - QoS 惩罚使用平方和（公平性增强），对偶更新仍使用线性 signed margin。
     - 输出指标包含：EE、qos_violation（均值）、qos_violation_sum（线性和）、qos_violation_sq_sum（平方和）、lambda、tx_power 等。

#### 2.4.2 `environment.py`

- 实现内容：兼容壳文件。
- 实现方式：仅 re-export `UbsRsmaEvn`、`MbPpoEnv`，把逻辑统一到 `mb_ppo_environment.py`。

#### 2.4.3 `channel_model.py`

- 实现内容：A2G/G2G 信道增益模型。
- 实现方式：
  - A2G：基于 LoS 概率 + 路损近似（返回大尺度增益）。
  - G2G：自由空间路损近似。

#### 2.4.4 `utils.py`

- 实现内容：环境工具。
- 实现方式：Jain 指数、obs/state 包装成张量。

## 3. 当前工程实现的关键完成项

1. 环境实现集中化：主逻辑统一在 `mb_ppo_environment.py`，降低历史分叉维护成本。
2. MHA 移除后网络稳定：保留多分支动作头，PPO/MB-PPO 通过 `single_head` 统一基线切换。
3. 连续动作到物理量映射闭环：轨迹、功率、速率、预编码均有明确映射路径。
4. 对偶约束机制可运行：`qos margin -> 归一化 -> lambda 更新 -> 下一轮奖励惩罚`，并支持 lambda 上限。
5. 训练与评估数据全链路可追踪：TensorBoard + pickle + checkpoint。
6. 对比绘图脚本可直接复用：支持 MB-PPO/PPO 四指标收敛图。
7. QoS 公平性强化：奖励侧采用平方违规量，违规大的用户受到更强惩罚。
8. URA+Phase 信道敏感性保留：为多分支轨迹/预编码协同优势提供物理可辨性。

## 4. 与论文草稿差异（代码现实 vs 草稿叙述）

以下差异基于 `多分支ppo.tex` 与当前代码对照：

### 4.1 网络结构差异

- 草稿：强调 MB-PPO，多分支结构（已删除 MHA 叙述）。
- 代码：`MultiBranchActorCritic` 使用 GT 编码 + 共享 MLP + 四分支输出，支持 `single_head` 退化为 PPO。
- 结论：主叙述与代码基本一致，细节上代码是 MLP 编码实现，不包含注意力模块。

### 4.2 信道/物理建模差异

- 草稿：偏完整 URA + Rician + 3D 几何与严格物理模型表达。
- 代码：
  - 环境主链路在 `mb_ppo_environment.py` 内保留 URA 导向矢量、复信道与莱斯叠加；
  - `_compute_rsma_rates` 使用显式 common/private SINR 和速率表达；
  - `channel_model.py` 仍保留大尺度参考模型（用于兼容与上界估计）。
- 结论：核心训练链路已与草稿中的 URA+Rician 方向对齐，仍保留工程化可调参数。

### 4.3 推进功耗模型差异

- 草稿：旋翼推进功率公式较完整。
- 代码：采用简化模型 `fly_power_base + fly_power_coeff * v^2`。
- 结论：训练稳定、实现简洁，但与理论公式存在简化差异。

### 4.4 奖励与目标差异

- 草稿：目标写作偏全局 EE 优化问题表述。
- 代码：按 step 计算 `EE - lambda * QoS_gap_sq_sum - power_penalty`，以 RL 长期回报近似求解。
- 结论：算法路径一致（约束优化思想一致），但求解形式是 RL 近似，不是显式解析优化。

### 4.5 QoS 约束执行差异

- 草稿：以约束形式给出 QoS 下界。
- 代码：
  - 计算 `qos_margin = R_th - R_k`；
  - 训练中用对偶变量 lambda 更新（默认使用归一化正部信号，避免多用户正负抵消）；
  - `lambda_decay_gate_ratio`/`lambda_decay_offset` 保留为兼容字段，不再参与当前主链路更新；
  - 使用 `lambda_max` 限幅避免数值爆炸；
  - 默认支持固定 R_th 强制对齐（paper_fixed_qos_threshold）。
- 结论：约束机制在代码中为 primal-dual 可训练实现，不是硬约束投影。

### 4.6 保密链路差异

- 草稿：不以窃听者/保密速率为主线。
- 代码：当前 MB-PPO 主链路已移除保密速率融合实现，不再走“历史安全速率 + 映射速率”叠加。
- 结论：该部分与当前草稿方向一致。

### 4.7 实验章节成熟度差异

- 草稿：仿真章节仍以叙述骨架为主。
- 代码：训练、扫参、绘图脚本齐全，可直接产生结果图与指标文件。
- 结论：工程端已经具备可复现实验流程，论文端仍需把配置表、统计口径、实验细节写全。

## 7. 公式与代码实现映射（新增）

本节给出论文关键公式在代码中的实现状态。结论采用三档：
- 已实现：代码中有直接对应计算。
- 近似实现：思路一致但做了工程化近似。
- 未实现：尚无直接实现。

### 7.1 RSMA 物理层与速率

1. 预编码重构
- 公式：将网络输出拆为实部/虚部形成复预编码向量。
- 状态：已实现。
- 代码：`_build_beams_and_powers`。

2. 公共流与私有流 SINR
- 公式：
  - $\gamma_c = \frac{P_c |h^H p_c|^2}{\sigma^2 + \sum_k P_k |h^H p_k|^2}$
  - $\gamma_k = \frac{P_k |h^H p_k|^2}{\sigma^2 + \sum_{j\neq k} P_j |h^H p_j|^2}$
- 状态：已实现。
- 代码：`_compute_rsma_rates`。

3. 速率表达
- 公式：
  - $R_c = B\log_2(1+\min_k\gamma_{c,k})$
  - $R_k = B\log_2(1+\gamma_k)$
  - $R_{k,tot}=R_k+\beta_k R_c$
- 状态：已实现。
- 代码：`_compute_rsma_rates`，并由 `rsma_mode` 切换 RSMA/SDMA/NOMA。

### 7.2 URA 空地莱斯信道

1. URA 导向矢量
- 公式：二维阵列相位推进形成 $a(\theta,\phi)$。
- 状态：已实现。
- 代码：`_ura_steering_vector`。

2. 莱斯信道叠加
- 公式：$h=\sqrt{\beta}(\sqrt{\frac{K}{K+1}}a_{los}+\sqrt{\frac{1}{K+1}}g_{nlos})$。
- 状态：已实现。
- 代码：`_channel_vector_complex`。

### 7.3 功率与可行域映射

1. MB-PPO 模式硬映射
- 公式：
  - $\rho=\frac{a_\rho+1}{2}$
  - $\alpha=\text{Softmax}(z)$
  - $P_s=\rho P_{max}\alpha_s$
- 状态：已实现。
- 代码：`_build_beams_and_powers(use_hard_mapping=True)`。

2. PPO 基线无硬映射
- 公式：不强制投影到严格可行域，通过奖励项惩罚超功率。
- 状态：已实现。
- 代码：`_physical_rate_mapping` 中 `use_hard_mapping = (baseline != "ppo")`。

### 7.4 奖励与对偶更新

1. 能效
- 公式：$EE=\frac{\sum_k R_k}{P_{fly}+P_{tx}}$。
- 状态：已实现（单位换算做了工程化处理）。
- 代码：`get_reward`。

2. QoS 违规量（公平性强化）
- 公式：
  - margin: $m_k=R_{th}-R_k$
  - gap: $g_k=\max(0,m_k)$
  - penalty: $\sum_k g_k^2$
- 状态：已实现。
- 代码：`get_reward`。

3. 奖励函数
- 公式：$r = w_{ee}EE - \lambda w_q\sum_k g_k^2 - w_p \cdot \text{power\_violation}$。
- 状态：已实现。
- 代码：`get_reward`。

4. 对偶变量更新
- 公式：
  - margin: $m_k = R_{th} - R_k$
  - 默认聚合（防抵消）：$\tilde g_k = \text{clip}(\max(0,m_k)/R_{th},0,1)$
  - 回合信号：$\bar g = \frac{1}{N}\sum_{n=1}^{N}\frac{1}{K}\sum_{k=1}^{K}\tilde g_k[n]$
  - 更新：$\lambda\leftarrow\text{clip}(\lambda+\eta\cdot\bar g,0,\lambda_{max})$
- 状态：已实现。
- 代码：环境 `get_reward` 生成 `qos_dual_signal_norm_for_lambda`（`lambda_signal_aggregation` 可切换 `positive_mean/signed_mean`），学习器 `update_lambda` 更新。

## 8. 结论

当前代码已经形成可运行、可训练、可评估、可绘图的 MB-PPO 工程闭环；并且关键公式（RSMA 速率、URA+Rician 信道、对偶更新）均可在代码中定位到直接实现。与论文草稿相比，当前差异主要集中在“理论最优表达 vs 工程可训练实现”的近似层面，而不是算法主干缺失。

## 10. 参数二次缩放与公式变换清单

本节专门记录“配置里设置了参数，但代码里又经过公式缩放/换算”的项目，避免实验结论被隐式数值变换误导。

1. 固定 QoS 阈值强制逻辑（R_th）
- 输入参数：`paper_fixed_qos_threshold`、`qos_threshold`、`enforce_fixed_qos_threshold`
- 代码换算：
  - 当 `enforce_fixed_qos_threshold=True` 时，训练使用 `paper_fixed_qos_threshold` 覆盖 `qos_threshold`
  - 运行时存在上界保护：`fixed_rth = min(fixed_rth, 0.12)`
- 代码位置：`run_mbppo.py::_apply_fixed_qos_threshold`

2. 发射功率上限：dBm -> W
- 输入参数：`tx_power_max_dbm`
- 代码换算：`tx_power_max_w = 1e-3 * 10^(tx_power_max_dbm/10)`
- 代码位置：`MbPpoEnv.__init__`

3. EE 奖励项：两级缩放
- 输入参数：`reward_ee_scale`
- 代码换算：
  - 环境初始化时先乘 `1000`：`self.reward_ee_scale = reward_ee_scale * 1000`
  - 奖励计算时再乘 `1e-6`：`ee_term = self.reward_ee_scale * EE * 1e-6`
- 等效：`ee_term = reward_ee_scale * EE * 1e-3`
- 代码位置：`MbPpoEnv.__init__`、`MbPpoEnv.get_reward`

4. QoS 罚项：平方和 + 对偶乘子
- 输入参数：`reward_qos_scale`、`lambda_penalty`
- 代码换算：`qos_term = lambda_penalty * reward_qos_scale * sum(max(0, R_th - R_k)^2)`
- 代码位置：`MbPpoEnv.get_reward`

5. QoS 对偶更新信号：归一化聚合 + 截断
- 输入参数：`lambda_signal_aggregation`（默认 `positive_mean`）
- 代码换算：
  - 正部聚合模式：`signal_k = clip(max(0, R_th - R_k) / R_th, 0, 1)`
  - 备选有符号模式：`signal_k = clip((R_th - R_k) / R_th, -1, 1)`
  - 回合信号取 step-level 用户均值再做回合均值
- 代码位置：`MbPpoEnv.get_reward`

6. 对偶变量更新：信号裁剪 + 上下界裁剪
- 输入参数：`lambda_lr`、`lambda_max`
- 代码换算：
  - 更新信号先裁剪到 `[-1, 1]`
  - `lambda <- clip(lambda + lambda_lr * signal, 0, lambda_max)`
- 代码位置：`MultiBranchPPOLearner.update_lambda`

7. 速率单位：Mbps <-> bps
- 代码中速率常以 Mbps 维护；计算 EE 时转回 bps：`throughput_mbps * 1e6`
- 代码位置：`_compute_rsma_rates`、`MbPpoEnv.get_reward`

8. 功率越限惩罚：归一化比例
- 输入参数：`reward_power_penalty_scale`
- 代码换算：`power_violation_ratio = max(0, P_tx - P_max) / P_max`
- 奖励项：`power_term = reward_power_penalty_scale * power_violation_ratio`
- 代码位置：`_physical_rate_mapping`、`MbPpoEnv.get_reward`

9. 观测中的速率与 QoS 特征：按理论上限或阈值归一化
- GT/UAV 速率特征除以 `achievable_rate_*_max`
- QoS gap 特征除以 `qos_threshold`
- 代码位置：`get_obs_agent`、`get_state`

10. 信道与噪声：dB 量纲到线性量纲
- 噪声谱密度：`n0 = 1e-3 * 10^(-200/10)`（W/Hz）
- 理论最大速率估计使用 `log2(1 + SNR)`，并乘 `1e-6` 记为 Mbps
- 代码位置：`UbsRsmaEvn.__init__`

4. 自适应引导增益：
  - 引导权重按最弱用户 QoS 缺口强度提升，缺口越大，引导越强。

## 15. 2026-04-15：QoS 违约不降零三根因修复（新增）

### 15.1 修复目标

围绕“QoS 违约量长期不降到 0”的三条根因，完成可复现实装：

1. 核心区惩罚过强导致目标失衡。
2. 核心区提前终止造成有效优化时域被截断。
3. 对偶信号按用户求均值时出现正负抵消。

### 15.2 代码改动清单

1. `mb_ppo_env/mb_ppo_environment.py`
  - 新增 `lambda_signal_aggregation`（`positive_mean/signed_mean`），默认 `positive_mean`。
  - 新增 `_build_lambda_signal_vector`，默认采用
    $\tilde g_k=\text{clip}(\max(0,R_{th}-R_k)/R_{th},0,1)$，避免用户间抵消。
  - 新增 core 惩罚退火：`core_penalty_warmup_episodes`、`core_penalty_start_scale`，并记录
    `core_penalty_factor`、`effective_core_penalty_scale`。
  - 新增 core 终止延迟启用：`core_terminate_start_episode`，`get_terminate` 改为 `_core_terminate_enabled()` 判定。
  - `info` 新增字段：`lambda_signal_aggregation`、`core_penalty_factor`、`effective_core_penalty_scale`、`core_terminate_enabled`、`core_terminate_start_episode`。

2. `algo/mb_ppo/mb_ppo_config.py`
  - 默认改为：
    - `lambda_signal_aggregation='positive_mean'`
    - `terminate_on_core_violation=False`
    - `core_terminate_start_episode=1000`
    - `core_penalty_warmup_episodes=1000`
    - `core_penalty_start_scale=0.15`
    - `reward_core_penalty_scale=22.0`（配合退火降低早期主导效应）

3. `algo/mb_ppo/run_mbppo.py`
  - `_build_env` 透传上述新参数。
  - CLI 新增：
    - `--lambda-signal-aggregation`
    - `--core-terminate-start-episode`
    - `--core-penalty-warmup-episodes`
    - `--core-penalty-start-scale`
  - 训练日志新增：`train/core_penalty_factor`、`train/effective_core_penalty_scale`、`train/core_terminate_enabled`。

4. `algo/offpolicy_baselines/run_offpolicy.py`
  - `_build_env_kwargs` 同步透传修复相关参数，确保 on/off-policy 环境语义一致。

### 15.3 理论到实现的对应关系（本次）

1. 根因一（可行域压力过大）
  - 理论：避免约束惩罚在训练早期压制主目标。
  - 实现：core penalty 线性退火。

2. 根因二（提前终止截断）
  - 理论：维持完整 rollout 时域以稳定策略梯度估计。
  - 实现：默认关闭 core 提前终止；如启用则延迟到指定回合后生效。

3. 根因三（对偶信号抵消）
  - 理论：单一对偶变量应与“任一用户违规都需被惩罚”的目标一致。
  - 实现：默认对偶信号改为正部聚合（可回切 `signed_mean` 做对照实验）。

### 15.4 验证口径

建议用以下三组曲线核查本次修复是否生效：

1. `qos_violation_sum_rollout_mean`（QoS 违规收敛趋势）。
2. `lambda_penalty` 与 `qos_signal_rollout_mean`（对偶动态一致性）。
3. `core_penalty_factor`、`effective_core_penalty_scale`、`core_terminate_enabled`（约束退火与终止策略是否按计划工作）。
